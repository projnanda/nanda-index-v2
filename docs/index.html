NANDA Index

A global switchboard for AI agent discovery — federated resolution for the agentic web.

NANDA Index stores one index record per organization and maps any identity — a domain, an email, or a URN — to the correct next discovery object: an AI Catalog, a DNS-AID path, an A2A Agent Card, or a personal agent card.

It is the first hop in a three-hop resolution chain:

Requester → NANDA Index → Registry / Agent Card Host → Agent Runtime
NANDA Index does not host agents. It tells you where to find them. Discovery is delegated to whichever system the organization already uses.
Enterprise SMB Individual Open URN identity

Core concepts

The index record

Every registered organization has exactly one IndexRecord. It carries just enough information to route a requester to the next hop — never the agent itself.

Four registration types

Typehosting_pathWhoHow it resolves (hop 2)
Enterprise RegistryregistryOrgs running their own nanda-registryGET registry_url/agents/<slug>
DNS-AIDdns-aidDomain-controlled discovery via DNSDNS-AID lookup using the data field
SMB Agent CardsmbSmall businesses on a shared hostFetch the card directly from registry_url
Personal AgentpersonalIndividuals, email identity (no domain)Fetch the card directly from registry_url
Signed records. Index and EntityOwner records are signed with Ed25519 over a canonical-JSON projection (keys sorted, no whitespace), so any implementation can verify a record byte-for-byte.

Architecture

The index is a thin, queryable mapping. A single row points a requester at the right downstream system:

┌───────────────────────────────────────────────────────────────┐
│                          NANDA Index                          │

│  org_id      media_type                    registry_url       │
│  ──────────  ─────────────────────────     ────────────────   │
│  acme-corp   application/ai-catalog+json   https://reg.acme   │
│  skyblue     application/vnd.dns-aid+json  (data field)       │
│  moonbakery  application/a2a-agent-card    https://host39.org │
│  john        application/a2a-agent-card    https://host39.org │
└───────────────────────────────────────────────────────────────┘

Resolution chain

Give the index a locator; it returns the next discovery object. Each hop narrows from identity → registry → agent card → live runtime.

# 1 — Resolve a locator to an IndexRecord
GET /api/v1/resolve?locator=urn:ai:domain:acme.com:agent:flights
   → IndexRecord { registry_url, identifier }

# 2 — Fetch the catalog entry from the org's registry
GET <registry_url>/agents/<identifier>
   → CatalogEntry { url }   (the facts URL)

# 3 — Fetch the A2A Agent Card
GET <catalogEntry.url>
   → A2A Agent Card { url }   (the runtime endpoint)

# 4 — Talk to the agent
POST <agentCard.url>/run
   → Agent response

Example response

curl "https://api.nandaindex.org/api/v1/resolve?locator=urn:ai:domain:acme.com:agent:flights"

{
  "locator": "urn:ai:domain:acme.com:agent:flights",
  "identifier": "flights",
  "index_record": {
    "org_id": "acme",
    "registry_url": "https://registry.acme.com",
    "media_type": "application/ai-catalog+json"
  }
}

IndexRecord

The wire shape returned for any registered organization.

interface IndexRecord {
  org_id:          string;
  display_name:    string;
  domain:          string | null;   // null for personal (email-identity) entries
  registry_url:    string | null;   // null for DNS-AID entries
  ttl_seconds:     number;          // 3600–604800 (default 86400 / 24h)
  status:          "pending" | "active" | "suspended";
  email_verified:  boolean;
  domain_verified: boolean;         // activation is gated on this
  created_at:      string;
  updated_at:      string;

  // Catalog fields (optional)
  identifier?:  string;             // URN, e.g. "urn:ai:domain:acme.com"
  media_type?:  string;
  description?: string | null;
  tags?:        string[];
  publisher?:   { identifier: string; displayName: string; identityType?: string };
  metadata?:    Record<string, unknown>;  // NANDA routing hints
  data?:        Record<string, unknown>;  // DNS-AID discovery data
}
Two-step activation. A new record starts as pending. Verifying the contact email sets email_verified but does not activate the record. The org only becomes active (and appears in the public index) after DNS TXT domain verification succeeds — see below.

The media_type declares what the next hop returns.

Identity URNs

Locators are URNs. The resolver parses these forms:

TypeFormatExample
Enterprise / orgurn:ai:domain:<domain>urn:ai:domain:acme.com
Enterprise / agenturn:ai:domain:<domain>:agent:<slug>urn:ai:domain:acme.com:agent:support
Personalurn:ai:email:<email>urn:ai:email:john@hotmail.com
Legacyurn:ai:<domain>:<slug>urn:ai:nasiko.com:ankit
Customany valid URNurn:ai:org.agntcy

For agent URNs the returned identifier is the slug; for email URNs it is the full URN.

Register an organization

Lifecycle

create (pending) → verify email → DNS TXT challenge → verify domain → active

Via the web UI

Go to nandaindex.org → sign in → Dashboard → New Organization. Pick a registration type, fill the form, verify your email, then complete domain verification. Your record goes live once the domain is verified.

Via the API

  1. Create an account

    curl -X POST https://api.nandaindex.org/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"you@example.com","password":"yourpassword"}'
# → 201 { "token": "eyJ..." }

  2. Register your organization (you become its admin)

    TOKEN="eyJ..."

curl -X POST https://api.nandaindex.org/api/v1/orgs \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{
    "org_id": "acme",
    "display_name": "Acme Corp",
    "hosting_path": "registry",
    "domain": "acme.com",
    "contact_email": "agents@acme.com",
    "registry_url": "https://registry.acme.com",
    "identifier": "urn:ai:domain:acme.com",
    "media_type": "application/ai-catalog+json",
    "description": "Acme enterprise AI Catalog.",
    "tags": ["enterprise","ai-catalog"],
    "publisher": {
      "identifier": "urn:ai:domain:acme.com",
      "displayName": "Acme Corp",
      "identityType": "dns"
    }
  }'
# → 201 IndexRecord (status: "pending")

    Validation: org_id is lowercase ^[a-z0-9][a-z0-9-]*[a-z0-9]$ (2–64 chars); domain is required for every path except personal; registry_url is required for every path except dns-aid; ttl_seconds is 3600–604800.

  3. Verify the contact email

    Open the link from the email, or call it directly. This sets email_verified — it does not activate the org.

    curl "https://api.nandaindex.org/api/v1/verify-email?token=<token>"

  4. Verify the domain

    Prove control of the domain via a DNS TXT record — this is what flips the org to active. See the next section.

DNS TXT domain verification

ACME dns-01-style proof of domain control. To prove you control acme.com, you publish a TXT record the index issued, then ask the index to check it.

  1. Request a challenge (admin only)

    curl -X POST https://api.nandaindex.org/api/v1/orgs/acme/domain-challenge \
  -H "Authorization: Bearer $TOKEN"

{
  "domain":       "acme.com",
  "record_name":  "_nanda-challenge.acme.com",
  "record_type":  "TXT",
  "record_value": "nanda-verify=<token>",
  "expires_at":   "2026-06-25T12:00:00.000Z"
}

  2. Publish the TXT record

    At your DNS provider, create a TXT record at _nanda-challenge.acme.com with value nanda-verify=<token>. Challenges are valid long enough to absorb DNS propagation.

  3. Verify & activate (admin only)

    curl -X POST https://api.nandaindex.org/api/v1/orgs/acme/verify-domain \
  -H "Authorization: Bearer $TOKEN"
# → 200 IndexRecord (status: "active", domain_verified: true)

    If the record isn't found yet the endpoint returns 400 DOMAIN_NOT_VERIFIED with the TXT values it actually observed — DNS can take time to propagate, so retry shortly.

Personal (email-identity) orgs have no domain to verify. Calling verify-domain on one returns 400 NO_DOMAIN.

Production deployment

Prerequisites

  • A VPS with ~2 GB RAM (1 GB works with swap — the Next.js build is memory-heavy)
  • Docker & Docker Compose
  • DNS A records: nandaindex.org and api.nandaindex.org → server IP
# 1. Clone and configure
git clone https://github.com/projnanda/nanda-index-v2
cd nanda-index-v2
cp .env.prod.example .env.prod   # fill in every value

# 2. Add 2GB swap (only needed on 1GB servers)
fallocate -l 2G /swapfile && chmod 600 /swapfile
mkswap /swapfile && swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab

# 3. Build & start
docker compose -f docker-compose.prod.yml --env-file .env.prod up --build -d

# 4. Verify
curl https://api.nandaindex.org/health
TLS is automatic. Caddy 2 fronts the stack and provisions/renews certificates for both hostnames — no manual certbot step.
